BookSlot is a small embeddable widget: a business drops one script tag on their site and gets a
booking calendar that writes into their own availability rules. The interesting decisions weren't in
the UI — they were in the data layer and in the parts of the platform I don't control.

## Why the data layer moved

The first version used a hosted Postgres instance. That worked, but a booking widget is mostly reads:
"what slots are free on this day" runs on every render, from every embed, and the write path is a
few rows a day. Paying connection-pool overhead for that shape felt wrong.

```ts
// A read-heavy path deserves a read-heavy store.
const slots = await db.execute({
  sql: 'select start_at, end_at from slots where day = ? and booked = 0',
  args: [day],
});
```

## What the platform forced

A few constraints shaped the design more than any design doc did:

- **Cron granularity.** Reminder emails wanted an hourly sweep. The Hobby plan allows one daily cron,
  so the sweep became a daily job with a wider window.
- **Embeds are third-party context.** Anything that assumed same-origin cookies had to go.
- **Cold email is a deliverability problem, not a code problem.** Sending is the easy part.

## Where it still leaks

Timezones. Always timezones. The widget renders in the visitor's zone, the availability rules are
authored in the business's zone, and the reminder job runs in UTC. Three zones, one row.

## What I'd do differently

Write the timezone model down before writing the schema. See the
[BookSlot repo](https://github.com/TaulantSela) for how the current one ended up.
